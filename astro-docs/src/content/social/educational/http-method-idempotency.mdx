    {`Idempotency is one of those HTTP concepts most developers have heard before, but its value becomes obvious in a very practical situation.

A client sends \`DELETE /users/42\`.
The resource might already be deleted.
But the client never receives the response.

What should the client do now?

If the origin server implemented \`DELETE\` correctly, the client can simply send the same request again.

That is what idempotency buys you.

It does **not** mean that nothing happens.
It means that repeating the same request keeps the **intended effect** the same.

So whether:
- the first request never reached the origin server
- the origin server processed it but the response got lost on the way back
- an intermediary timed out and the client cannot tell what happened

...the client can retry the \`DELETE\` request and still converge on the same result: the resource is absent.

That is the real lesson:
- use the capabilities the HTTP protocol already gives you
- let clients rely on method semantics instead of guesswork
- if you implement the origin server, respect the protocol and implement HTTP methods properly

A lot of problems already have protocol-level answers if we actually use HTTP as designed.

Docs: https://thymian.dev/introduction/http-conformance/

Where have you seen idempotency make retry behavior easier to reason about?`}

Idempotency is not an academic term. It solves a very practical API problem.

A client sends DELETE for a resource and never receives the response. Did the request get lost? Did the origin process it? Should the client retry?

If DELETE is implemented according to HTTP semantics, the answer is straightforward: retry it.

Idempotent does not mean “nothing happens”. It means repeating the same request preserves the intended effect.

That matters when:
✅ the request is lost before it reaches the origin
✅ the origin deletes the resource but the response is lost on the way back
✅ an intermediary times out and the client cannot tell what happened

In all of those cases, the client can send DELETE again and converge on the same state: the resource is absent.

That is one of the big benefits of using HTTP as designed:
→ the protocol already contains recovery semantics for common failure scenarios
→ clients can rely on method behavior instead of inventing custom retry rules
→ origin servers need to respect the method semantics for this to work

A lot of problems already have protocol-level answers if we actually use HTTP as designed.
