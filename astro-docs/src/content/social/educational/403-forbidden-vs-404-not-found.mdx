    {`When a user requests a resource they are not allowed to access, most APIs return \`403 Forbidden\`. Seems correct.

But that response confirms the resource exists. An attacker can use this to enumerate what is behind your API.

Example:

\`\`\`
GET /users/42/settings → 403 Forbidden
GET /users/43/settings → 404 Not Found
\`\`\`

Now the attacker knows user 42 exists and user 43 does not. Repeat this a few thousand times and you have a full user directory.

RFC 9110 (Section 15.5.4) addresses this directly. It says an origin server that wants to hide the current existence of a forbidden target resource MAY respond with \`404 Not Found\` instead.

This is not a workaround. It is a deliberate protocol design choice. The RFC authors anticipated this exact scenario.

When to use which:

- **\`403 Forbidden\`** — when the client already knows the resource exists (e.g., they own it but lack a specific permission), or when hiding existence does not matter.
- **\`404 Not Found\`** — when revealing existence itself is sensitive. User profiles, admin endpoints, internal resources, tenant-specific data.

Thymian can surface these kinds of protocol-level semantics for you and your agents across the entire lifecycle — while linting your API description, testing implementations, and analyzing real traffic.

Docs: https://thymian.dev/introduction/http-conformance/

Do your APIs use 404 to hide resources, or do you always return 403?`}
