    {`A status code is not just a short label for “worked” or “failed”.

In HTTP, the status code tells clients what kind of response they received and what they can reasonably do next. That makes status codes part of your API behavior.

A few examples:

- \`202 Accepted\` means the request was accepted, but processing has not completed yet. Clients should not assume the work is done.
- \`303 See Other\` points the client to another resource that can be retrieved with \`GET\`.
- \`422 Unprocessable Content\` says the content is syntactically valid, but semantically invalid for the operation.

Those differences matter because clients, caches, gateways, SDKs, generated docs, monitoring systems, and agents all read status codes as machine-readable behavior.

If an API returns \`200 OK\` with an error object, humans might understand it. Generic HTTP tooling will not.

If an API returns \`500 Internal Server Error\` for validation problems, clients cannot distinguish “retry later” from “fix your request”.

If an API returns \`200 OK\` for asynchronous work that is merely accepted, clients lose the standard \`202 Accepted\` signal.

Good status codes make APIs easier to automate, test, cache, monitor, and reason about.

Docs: https://thymian.dev/introduction/http-conformance/

    {`HTTP status codes are API behavior, not labels.

It is tempting to treat status codes as a small set of buckets:

- 2xx = success
- 4xx = client problem
- 5xx = server problem

That is a useful start, but it misses the point.

In HTTP, the status code defines the semantics of the response. It tells clients, caches, gateways, generated SDKs, monitoring systems, and agents what happened and what they can reasonably do next.

A few examples:

→ \`202 Accepted\`
The request was accepted, but processing is not complete. The client should not treat the operation as finished.

→ \`303 See Other\`
The response points the client to another resource that can be retrieved with \`GET\`.

→ \`422 Unprocessable Content\`
The request content is syntactically valid, but semantically invalid for the operation.

These distinctions are not cosmetic. They affect retries, error handling, caching, documentation, SDK behavior, observability, and automation.

Examples of weak API behavior:

- returning \`200 OK\` with an error object
- returning \`500 Internal Server Error\` for validation problems
- returning \`200 OK\` for asynchronous work that should be \`202 Accepted\`
- collapsing distinct client-side problems into generic errors, even when \`422\` would be more precise

Good status code usage makes APIs easier to consume because the protocol already carries useful machine-readable meaning.

That is why HTTP conformance is not just about being “technically correct”. It is about making API behavior explicit and reliable.

Learn more: https://thymian.dev/introduction/http-conformance/
