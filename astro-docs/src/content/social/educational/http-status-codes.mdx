  <PostText platform="reddit" label="Post body">
    {`Most developers know the basic rule: don't return \`200 OK\` for a failed request.

But HTTP status codes do much more than label success vs error. In HTTP, the status code is the main source of the response semantics.

A few examples:

- \`404 Not Found\` vs \`410 Gone\`
  \`404\` means the resource is not found.
  \`410\` means it existed before and is intentionally gone.

- \`403 Forbidden\` vs \`404 Not Found\`
  RFC 9110 explicitly allows an origin server to respond with \`404\` instead of \`403\` if it wants to hide the existence of the resource.

Those are not cosmetic differences. They change what clients, caches, and other tooling can infer from the response.

This is exactly the kind of HTTP nuance Thymian can surface across the entire lifecycle. It gives you and your agents hints about protocol capabilities like these distinctions — and about violations in status code usage — while linting your OpenAPI description, testing implementations, and analyzing real traffic.

\`\`\`bash
npx thymian lint --spec openapi:openapi.yaml
npx thymian test --spec openapi:openapi.yaml
npx thymian analyze --traffic har:./traffic.har
\`\`\`

Docs: https://thymian.dev/introduction/http-conformance/

Which status code distinction do you see misused most often?`}
  </PostText>
