Hey r/webdev 👋

We just shipped **Thymian**, a free open-source tool that brings HTTP conformance testing to your API workflow.

The core idea: write a rule once, then use it across three modes:

- `thymian lint` — static analysis of your OpenAPI spec
- `thymian test` — runtime validation against a live server
- `thymian analyze` — analysis of recorded HAR traffic from production

No more "it passes lint but breaks in production." One rule, three validation layers.

Quick start (no install needed):

```bash
npx thymian lint --spec openapi:openapi.yaml
```

Happy to answer any questions. What HTTP edge cases do you run into most in production?

🔗 https://thymian.dev

<PlatformSection platform="linkedin">

We built Thymian because we kept seeing the same problem: APIs that pass static linting but fail in production due to subtle HTTP non-conformance.

The root cause? Most teams validate their API at one layer only — either the spec, or a live test, or production traffic. But HTTP bugs often only surface in one specific context.

**Thymian solves this with a "Write Once, Validate Everywhere" model:**

✅ `thymian lint` — catches spec violations before you ship
✅ `thymian test` — validates your running service against real HTTP semantics
✅ `thymian analyze` — scans recorded production traffic (HAR files) for violations

The same rule runs across all three modes. No duplication. No gaps.

It's open-source, zero-install for quick runs (`npx thymian`), and extensible via plugins.

If you've ever dealt with cache poisoning, incorrect Content-Type handling, or missing RFC 9110 compliance — this is built for you.

🔗 https://thymian.dev
