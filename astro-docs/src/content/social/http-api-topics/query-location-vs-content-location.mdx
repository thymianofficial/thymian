    {`Last week, we talked about the QUERY method. And since we love diving deep into things, we want to take a closer look today.

When a \`QUERY\` request succeeds with a 2xx response, that response can include both \`Location\` and \`Content-Location\`.

So the question is: what is the difference between them?

This is one of those HTTP details that can feel confusing because the header names sound almost interchangeable. They are not. In the context of a successful \`QUERY\` response, they point to two different ideas.

**The breakdown:**

- 🔁 **\`Location\` = the query itself**
  Think of this as a stored query. If you later send a \`GET\` request to that URI, the server can re-execute the search/filter live and return fresh, up-to-date data.

- 📌 **\`Content-Location\` = this specific result**
  Think of this as a stored result or snapshot. If you later send a \`GET\` request to that URI, you get the data state from that specific point in time, without recalculating the query.

Example:

\`\`\`http
QUERY /products HTTP/1.1
Content-Type: application/json

{"color":"red","type":"shoes","maxPrice":50}
\`\`\`

A successful response could say:

\`\`\`http
HTTP/1.1 200 OK
Location: /queries/red-shoes-under-50
Content-Location: /query-results/2026-06-26T10-15-00Z
\`\`\`

**The analogy: shoe shopping**

- 🔁 **\`Location\`** is a bookmarked search link for "red shoes under $50". If you click it tomorrow, you might see new shoes that were just added to the store.

- 📌 **\`Content-Location\`** is a snapshot of the exact list of shoes you saw at that specific second. If you open it tomorrow, it shows the exact same historical results.

**Key takeaway:** \`Location\` points to something you can run again. \`Content-Location\` points to what you got this time.

Have you started experimenting with the \`QUERY\` method in your projects yet?
