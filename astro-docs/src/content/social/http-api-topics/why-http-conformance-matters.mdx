Your API can return `200 OK` in every test and still be broken in production. Here's why.

HTTP conformance is the set of rules defined in RFC 9110 that every HTTP participant — clients, proxies, CDNs, gateways — is expected to follow. When your API violates these rules, the failure mode isn't a test failure. It's a CDN that doesn't cache what it should, a mobile client that re-authenticates in an infinite loop, or a browser that ignores your CORS headers.

These failures are:
- **Silent** — no error is thrown, the API "works"
- **Environment-specific** — works fine against your test server, breaks with the CDN in front
- **Intermittent** — the CDN only caches certain responses, so failures are hard to reproduce

The fix isn't more unit tests. It's conformance testing — checking your API's behavior against the HTTP specification itself.

That's what Thymian does. Same rules work on your OpenAPI spec (before you ship), against your live server (in CI), and against real traffic from production (via HAR analysis).

**One of the most insidious classes of API bugs is invisible to your test suite.**

Your API returns `200 OK`. Tests pass. CI is green. You deploy.

Then:
- Your CDN caches a response it shouldn't (or doesn't cache one it should)
- Mobile clients enter an authentication retry loop
- Browser CORS preflight requests silently fail
- Downstream services interpret error responses incorrectly

The root cause? HTTP non-conformance — your API violates the semantics defined in RFC 9110 in subtle ways that unit tests and integration tests almost never catch, because those tests talk directly to your server, not through the full HTTP stack.

This is why conformance testing is a distinct testing discipline from integration testing. You're not testing your business logic. You're testing your protocol behavior.

Thymian addresses this by running the same conformance rules across three contexts:
- Static analysis of your OpenAPI spec
- Runtime testing against a live server
- Analysis of recorded production traffic

The spec violation you catch with `thymian lint` is the same one that would have caused a CDN caching anomaly in production.
