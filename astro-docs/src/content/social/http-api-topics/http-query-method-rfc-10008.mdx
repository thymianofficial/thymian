    {`Have you ever used \`POST /search\` for a read-only operation because the query was too large or too complex for a URL?

That is the problem the new HTTP \`QUERY\` method solves.

RFC 10008 was just released in June 2026 and standardizes a method for safe, idempotent requests with request content.

The gap it fills is pretty specific:

- \`GET\` is safe and idempotent, but the query input usually has to live in the URI.
- \`POST\` can carry request content, but it does not tell generic HTTP tooling that the operation is safe or idempotent.
- \`QUERY\` carries request content like \`POST\`, while explicitly being safe and idempotent like \`GET\`.

Example:

\`\`\`http
QUERY /feed HTTP/1.1
Host: example.org
Content-Type: application/x-www-form-urlencoded

q=foo&limit=10&sort=-published
\`\`\`

That matters because HTTP semantics are used by more than your application code. Retries, caches, gateways, observability tools, generated clients, and security infrastructure all make decisions based on method semantics.

A few important details from the RFC:

- \`Content-Type\` is required and must match the query content.
- Responses to \`QUERY\` are cacheable, but cache keys must include the request content and related metadata.
- \`Accept-Query\` lets a resource advertise supported query media types.
- \`Location\` and \`Content-Location\` can point clients to resources that can later be retrieved with \`GET\`.
- Browser CORS preflight applies because \`QUERY\` is not a CORS-safelisted method.

It is not “GET with a body.” It is a separate method for safe, idempotent server-side query processing.

Curious how quickly clients, frameworks, gateways, and CDNs will adopt it.

RFC: https://www.rfc-editor.org/info/rfc10008`}

    {`**What should an API use when a read-only query is too complex for a URL, but \`POST\` communicates the wrong semantics?**

That is the problem solved by HTTP \`QUERY\`.

RFC 10008 was just released in June 2026 and defines a standard method for safe, idempotent requests with request content.

This solves a common API design mismatch:

- \`GET\` has clear safe/idempotent semantics, but complex query input often does not fit well in a URI.
- \`POST\` accepts request content, but generic HTTP infrastructure cannot assume the operation is safe or idempotent.
- \`QUERY\` accepts request content while explicitly being safe and idempotent.

Example:

\`QUERY /feed\`
\`Content-Type: application/json\`

That semantic distinction matters. HTTP method semantics influence automatic retries, caching behavior, intermediaries, client generation, observability, and security tooling.

✅ \`Content-Type\` is required and must match the query content
✅ \`QUERY\` responses are cacheable, with cache keys incorporating request content and metadata
✅ \`Accept-Query\` lets resources advertise supported query formats
✅ \`Location\` and \`Content-Location\` can connect \`QUERY\` results to later \`GET\` requests
✅ CORS preflight applies because \`QUERY\` is not CORS-safelisted

The important nuance: \`QUERY\` is not “GET with a body.” It is its own HTTP method for safe, idempotent server-side query processing.

Adoption will take time across frameworks, clients, gateways, and caches — but the semantics are now standardized.
