  <PostText platform="reddit" label="Post body">
Cache poisoning via header injection is still one of the most underestimated attack vectors in modern APIs. The attacker exploits the gap between what your origin server trusts and what your CDN caches.

A classic setup: your app blindly reflects custom headers (`X-Forwarded-Host`, `X-Original-URL`) into cached responses. An attacker crafts a request with a malicious value — and the CDN caches the poisoned response for everyone.

With Thymian, you can write a lint rule that flags any response that reflects an unvalidated inbound header:

```typescript
import { httpRule } from '@thymian/core';

const DANGEROUS_HEADERS = ['x-forwarded-host', 'x-original-url', 'x-rewrite-url'];

export default httpRule('no-reflected-cache-headers')
  .severity('error')
  .match(({ request, response }) =>
    DANGEROUS_HEADERS.some(
      (h) => request.headers[h] && response.headers['cache-control']?.includes('public'),
    ),
  )
  .report(() => ({
    message: 'Response is publicly cached while reflecting a potentially attacker-controlled header.',
    docs: 'https://thymian.dev/guides/http-rules/how-to-use-rules/',
  }));
```

Run this against your HAR files from production traffic:

```bash
npx thymian analyze --har ./prod-traffic.har --spec openapi:openapi.yaml
```

It'll surface any request/response pair where this pattern fires — before an attacker does.

<PlatformSection platform="linkedin">

**Cache poisoning is still one of the most underestimated API attack vectors — and most teams only discover it after an incident.**

The pattern is deceptively simple:

1. Attacker sends a request with a crafted header (`X-Forwarded-Host`, `X-Original-URL`)
2. Your origin reflects the value into the response (common in reverse proxy setups)
3. Your CDN caches the poisoned response and serves it to all subsequent users

The fix isn't just "don't reflect headers." You need systematic detection across your real traffic.

This is exactly what Thymian's analyze mode is built for. Point it at a HAR file from your CDN logs:

```bash
npx thymian analyze --har ./prod-traffic.har
```

A custom rule flags every request/response pair where an attacker-injectable header appears alongside a `Cache-Control: public` response — before it becomes a vulnerability in the wild.

Thymian rules work across lint (spec), test (live server), and analyze (traffic) — so you're protected at every layer.
