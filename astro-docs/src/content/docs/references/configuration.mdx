This page is the canonical reference for Thymian configuration.

## Where configuration is loaded from

Thymian resolves configuration in this order:

<Steps>

1. If you pass `--config <path>`, Thymian uses that file.
2. Otherwise, Thymian looks for `thymian.config.yaml` in `--cwd` (or your current directory).
3. If no file is found, Thymian uses built-in defaults.

</Steps>

<Aside type="note">
  Thymian only checks `thymian.config.yaml` by default. If you want another filename or extension, pass it explicitly with `--config`.
</Aside>

## Full configuration schema

The full configuration schema reference is auto-generated from the config JSON schema.

See [Configuration Schema](/references/configuration-schema/) for the complete list of supported keys, types, and defaults.

### `rules` entry shape

Each key in `rules` is a rule ID. The value can be either:

- A **severity string** (`off | error | warn | hint`) to override the rule's default severity.
- A **rule config object** with the following fields:
  - `severity` (`off | error | warn | hint`, optional) — override the rule severity.
  - `type` (`string[]`, optional) — restrict the rule to specific rule types (e.g., `['request', 'response']`).
  - `skipOrigins` (`string[]`, optional) — list of origin identifiers to ignore for this rule. When a result originates from one of these origins, it is suppressed.
  - `options` (`object`, optional) — rule-specific options.

Example:

```yaml
rules:
  rfc9110/status-code-semantics: warn
  rfc9110/content-type-header:
    severity: error
    skipOrigins:
      - 'GET /legacy-endpoint'
    options:
      allowWildcard: false
```

### `specifications[]` item shape

Each entry supports:

- `type` (`string`) - for example `openapi`
- `location` (`string` or supported path source)
- `options` (`object`, optional)

Example:

```yaml
specifications:
  - type: openapi
    location: ./openapi.yaml
```

### `traffic[]` item shape

Each entry supports:

- `type` (`string`) - for example `fixture`
- `location` (`string` or supported path source)
- `options` (`object`, optional)

Example:

```yaml
traffic:
  - type: fixture
    location: static
```

### `plugins` item shape

Each plugin entry supports:

- `path` (`string`, optional) - explicit module/file path
- `verbose` (`boolean`, optional)
- `options` (`object`, optional)

Example:

```yaml
plugins:
  '@thymian/plugin-openapi': {}
  '@thymian/plugin-reporter':
    options:
      formatters:
        text: {}
```

## Precedence and overrides

After loading config, CLI flags can override parts of it. Common overrides include:

- `--spec <type:path>` overrides `specifications`
- `--traffic <type:path>` overrides `traffic`
- `--target-url <url>` overrides `targetUrl`
- `--rule-set <name>` extends `ruleSets` at runtime (appends and deduplicates)
- `--rule-severity <off|error|warn|hint>` overrides `ruleSeverity`
- `--log-level <trace|debug|info|warn|error|silent>` overrides `logLevel`
- `--debug` effectively overrides `logLevel` to `debug`
- `--verbose` effectively overrides `logLevel` to `info`
- `--autoload/--no-autoload` controls plugin autoloading

## Generated baseline config

`thymian generate config` creates a config based on built-in defaults plus your selected specification(s).

Typical generated baseline:

```yaml
specifications:
  - type: openapi
    location: ./openapi.yaml

traffic: []

ruleSets:
  - '@thymian/rules-rfc-9110'
  - '@thymian/rules-api-description-validation'

ruleSeverity: error
rules: {}

plugins:
  '@thymian/plugin-http-linter': {}
  '@thymian/plugin-openapi': {}
  '@thymian/plugin-request-dispatcher': {}
  '@thymian/plugin-sampler': {}
  '@thymian/plugin-reporter':
    options:
      formatters:
        text: {}
  '@thymian/plugin-http-tester': {}
  '@thymian/plugin-http-analyzer': {}
  '@thymian/plugin-har': {}
```

## Workflow-specific examples

### Static linting only

```yaml
specifications:
  - type: openapi
    location: ./openapi.yaml

ruleSets:
  - '@thymian/rules-rfc-9110'

plugins:
  '@thymian/plugin-http-linter': {}
  '@thymian/plugin-openapi': {}
  '@thymian/plugin-reporter':
    options:
      formatters:
        text: {}
```

```yaml
specifications:

    location: ./openapi.yaml

targetUrl: http://localhost:3000
```

## Validation behavior and failure modes

If config parsing or schema validation fails, the CLI exits with code `2` and prints an error.

Common issues:

1. Unsupported file extension when loading with `--config`
2. Invalid YAML/JSON syntax
3. Invalid value types (for example non-array `ruleSets`)
4. Missing specification with commands that require one

## Practical tips

- Prefer `thymian generate config` to bootstrap a correct baseline.
- Keep plugin configuration minimal first, then add plugin `options` incrementally.
- Use `--spec` for one-off checks without editing files.
- Keep `ruleSeverity: error` in CI unless you explicitly want warnings/hints to gate builds.
